Develop a SentinelDesk News Operations & Settlement system for an on-prem newsroom that ingests multi-source news feeds, cleans and deduplicates stories, and manages internal licensing charges and refunds without relying on the public internet. In the React web console, Editors upload feed files (XML/JSON/CSV) or paste batches of URLs, then review a queue where the system highlights normalized URLs, near-duplicates, and suspicious anomalies with clear inline explanations and side-by-side version diffs. When a new update matches an existing story, the interface prompts for a merge strategy (replace, append, or keep both) and captures a mandatory change note for traceability. Finance Reviewers work from a “Transactions” workspace to approve internal charges (for example, $25.00 per licensed story bundle) and process refunds or partial refunds tied to a specific story version; freezes can be applied to disputed transactions until an Auditor releases them. Administrators manage roles and permissions, configure rate limits per user, and run searchable audit reports with CSV export filtered by date range (MM/DD/YYYY), user, and action type, so key actions like merges, approvals, and refunds are quickly traceable.

On the system side, the NestJS backend exposes versioned REST APIs (v1, v2) consumed by the React client and publishes a local OpenAPI specification for offline development and QA. PostgreSQL persists stories, story versions, dedup clusters, cleansing events, fund ledgers, refund cases, and immutable audit logs; sensitive fields (user secrets, financial account identifiers, and any stored vendor contacts) are encrypted at rest, and all UI views apply server-side masking rules based on role. Security is enforced with CSRF tokens, strict output encoding to reduce XSS risk, parameterized queries to prevent SQL injection, and centralized NestJS guards/interceptors for permission checks and consistent data redaction. Authentication is strictly local username-and-password with a minimum 12 characters, lockout after 5 failed attempts for 15 minutes, session expiration after 30 minutes idle and 12 hours absolute, and optional offline MFA using time-based one-time passwords. Payment “gateway aggregation” is implemented as pluggable internal payment channels (prepaid balance, invoice credit, and purchase-order settlement), each requiring signed request payloads with timestamp nonces; the backend verifies signatures, rejects replays older than 5 minutes, and enforces idempotency so duplicate callbacks or tampered amounts cannot post twice. Reliability is handled via a local Redis cache for hot reads (5-minute TTL), per-user rate limiting (60 requests per minute), and an on-host job queue for reconciliation, notification banners, and nightly backups; backups run at 2:00 AM with 30-day retention and a tested restore procedure targeting a 2-hour recovery window, while local metrics/logs/traces are retained for 14 days with an in-app alerts dashboard. Deduplication uses URL normalization plus content fingerprints (SimHash/MinHash) with configurable similarity thresholds, and every cleansing, repair, and merge operation writes a traceable log entry so data corrections are auditable end-to-end.
